# Concept-Before-Use Rule（概念導入順序の原則）

**Extracted:** 2026-02-24
**Context:** Zenn 技術記事で初出の概念がアーキテクチャ図やコード内に説明なく現れ、読者が面食らう問題

## Problem

技術記事を書く際、筆者にとって自明な概念（ツール名、自作用語、設計パターン）を
説明なしにアーキテクチャ図やコードブロックに登場させてしまう。読者は「これは何？」と
なり、記事の信頼性と可読性が下がる。

典型的な発生パターン:
- 知名度の低いツール（例: Kimi K2.5）を読者が知っている前提で書く
- 自作の概念（例: spec.md）をアーキテクチャ図に出してから後で説明する
- 略語や専門用語を初出時に定義しない

## Solution

### 1. 未知ツールの導入テンプレート

読者の大半が知らないツールを扱う場合、以下の順序で導入する:

1. **何者か**（開発元、カテゴリ、一言説明）
2. **どのくらい凄いか**（ベンチマークスコア、他ツールとの比較表）
3. **独自の特徴**（差別化ポイントの説明）
4. **設計上の着眼点**（強み・弱みの分析 → 自分のユースケースへの橋渡し）

```markdown
## [ツール名] という選択肢

[ツール名] は [開発元] が開発した [カテゴリ] です。[一言説明]。

| ベンチマーク | [ツール] | [比較対象A] | [比較対象B] |
|---|---|---|---|
| [指標1] | xx% | yy% | zz% |

しかし [ツール名] の真の差別化ポイントは [独自機能] です。
[独自機能の説明 2-3文]

### 設計の着眼点: [強み] と [弱み]
[ベンチマークと機能から読み取れる強みと弱みの分析]
→ [自分の設計判断への橋渡し]
```

### 2. 自作概念の導入ルール

自分で名付けた概念（spec.md、ディスパッチ等）は、**使う前に定義する**:

- アーキテクチャ図に `spec.md` を出すなら、その前に「spec.md とは何か」を説明
- 初出時に太字 + 1-2文の定義を添える
- 読者の「これは何？」を図の前で解消しておく

### 3. チェックリスト（セルフレビュー用）

記事の各図・コードブロックについて:
- [ ] 図中の全要素は、それ以前の本文で説明済みか？
- [ ] 初出のツール名に「何者か」の説明があるか？
- [ ] 自作用語に定義があるか？
- [ ] 略語は初出時にフルネームを併記しているか？

## When to Use

- 技術記事にアーキテクチャ図やフロー図を含む場合
- 知名度の低いツールやサービスを記事の主題にする場合
- 自作の概念名・用語を導入する場合
- editor エージェントのレビューで "Audience Appropriateness" が C+ 以下の場合
